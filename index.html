Security Design

🔒
No public endpoints
All services use ClusterIP. No LoadBalancer or NodePort exposed to the internet.
🔑
Secrets management
Database credentials stored as Kubernetes Secrets, never in manifests or environment args.
🛡️
Least privilege
appuser has SELECT/INSERT/UPDATE/DELETE on appdb only. monitor user is read-only.
📦
Resource limits
CPU and memory requests/limits set on all containers. Prevents noisy-neighbour issues and OOMKill cascades.
❤️
Health probes
Liveness and readiness probes on both nginx and MariaDB. Kubernetes restarts unhealthy pods automatically.

Design Decisions

Why ClusterIP only?
Exposing services via LoadBalancer adds a public attack surface. Internal-only services are accessed via kubectl port-forward — authenticated through the AKS API server, which logs all access.
Why 2 Nginx replicas?
Demonstrates pod scheduling across nodes, rolling update behavior, and how Kubernetes routes traffic via kube-proxy. In production, a HorizontalPodAutoscaler would manage replica count based on CPU load.
Why managed-csi for storage?
Azure Disk CSI is the default StorageClass in AKS. It provisions Azure Managed Disks automatically and supports volume snapshots — relevant for backup and DR scenarios.
